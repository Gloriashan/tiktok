# 直播竞拍系统（VibeCoding 项目） —— 面试准备文档

---

## 一、业务背景

**项目名称**：直播竞拍系统（Live Auction System）

**业务模式**：这是一个 **「直播 + 实时竞拍」** 结合的电商平台。商家在直播间里展示商品，用户通过实时出价来竞拍购买，而不是传统电商的固定价格购买模式。

**为什么做这个方向？**

1. **市场趋势**：直播电商已经是主流（抖音直播带货、快手电商），但绝大多数直播电商是「固定价格 + 抢购」模式。而「直播竞拍」是一个细分的、有差异化的模式，在艺术品、二手奢侈品、收藏品、特殊商品等领域有真实需求。
2. **技术挑战性**：相比传统电商，直播竞拍需要处理 **实时通信（WebSocket）、高并发出价、倒计时一致性、分布式锁、数据最终一致性** 等一系列难度较高的技术问题，非常适合展示技术能力。
3. **可落地性**：项目范围可以控制，MVP 阶段不需要仓储物流、支付牌照等重基础设施，非常适合小型团队或个人快速验证。

---

## 二、目标用户

这个平台有两个核心角色：

### 1. 商家（Merchant）

- **身份**：拥有商品货源的个人或商家，需要在平台上进行直播拍卖
- **核心需求**：
  - 管理商品（上架、下架、编辑）
  - 创建和管理竞拍（设置起拍价、加价幅度、封顶价、时长）
  - 开播/下播，管理直播场次
  - 标记讲解中的商品，引导用户当前重点
  - 查看成交订单并处理发货
- **痛点**：传统拍卖行门槛高、流程慢；直播平台没有灵活的竞拍工具

### 2. 普通用户 / 买家（User）

- **身份**：在平台上浏览直播并参与竞拍的消费者
- **核心需求**：
  - 浏览正在直播的商家列表
  - 进入直播间观看实时竞拍
  - 对感兴趣的商品出价
  - 查看自己的出价状态（领先/被超过）
  - 支付成交订单，跟踪物流
- **痛点**：想买稀缺商品但抢不到；希望以有竞争力的价格获得心仪商品

## 三、真实问题（我们在解决什么）

### 核心问题
**如何让一个小商家像专业拍卖行一样，在直播间里高效、公平、实时地完成商品竞拍，同时保证出价过程的实时性和数据一致性。**

### 拆解为子问题

| 问题 | 具体表现 | 技术挑战 |
|------|---------|---------|
| **实时出价** | 用户出价后，所有直播间观众要立刻看到新价格和排行榜 | WebSocket 广播、延迟竞争条件 |
| **高并发防超卖** | 同一时间多人出价，不能出现双方都以为自己出价成功 | 分布式锁 + 数据库悲观锁 |
| **倒计时一致性** | 服务端重启后倒计时不能中断，各客户端倒计时需同步 | Redis 持久化倒计时 + 定时检查 |
| **竞拍到期自动处理** | 到时间自动成交/流拍，保证无人能加时出价 | 定时器（每秒检查）+ 事务 |
| **出价排行榜** | 实时展示前 N 名出价者和金额，且不能太慢 | Redis ZSet 有序集合 |
| **订单履约** | 竞拍成功后用户未支付、商家不发货等问题 | 超时自动取消 + 状态机管理 |
| **状态一致性** | Redis 缓存和 MySQL 数据库间的数据一致 | Redis 优先 + MySQL 兜底降级 |

## 四、限制条件

作为 VibeCoding 项目，我们面临以下真实约束：

| 限制 | 具体说明 |
|------|---------|
| **时间极紧** | 整个项目在 **2-3 周内** 由单人完成（从规划到 MVP），平均每天只能投入少量时间 |
| **单人开发** | 没有前后端分工，没有专职测试/运维/设计，一个人同时做前端 + 后端 + 数据库 + 部署 |
| **无专业UI设计** | 前端 UI 依赖 Ant Design 组件库，没有定制设计稿 |
| **无测试团队** | 几乎没有自动化测试，主要靠手动测试验证关键路径 |
| **无真实支付渠道** | 支付功能是模拟的（选项包括 wechat/alipay/bank 但无真实对接） |
| **无运维基础设施** | 没有 Docker/K8s，没有 CI/CD，没有监控告警，没有灰度发布 |
| **本地环境部署** | 依赖本机 MySQL + Redis，没有云端部署保障可用性 |
| **无真实视频流** | 直播视频流用假数据模拟，没有对接第三方直播云服务 |
| **预算为零** | 没有云服务费用，没有第三方 API 采购预算 |

这些限制决定了我们**必须做大量的取舍**，只做最关键的功能，优先保证核心链路跑通。

## 五、方案取舍（做了什么 vs 没做什么，为什么）

### ✅ 5.1 实现了的核心功能（优先级最高）

| 功能 | 优先级 | 为什么做了 |
|------|--------|-----------|
| **商家注册/登录 + JWT 鉴权** | P0 | 系统最基础的身份识别，没有就无法区分商家和用户 |
| **商品管理（CRUD + 图片上传）** | P0 | 竞拍的前提是要有商品，必须让商家能上架商品 |
| **竞拍创建/开始/取消** | P0 | 核心业务实体，商家需要能设定竞拍参数 |
| **用户实时出价** | P0 | 整个系统的核心交互，必须保证出价能实时到达 |
| **竞拍倒计时 + 自动成交** | P0 | 倒计时不准 = 竞拍无效，必须精确到秒级 |
| **Redis 倒计时/排行榜/分布式锁** | P0 | 支撑高并发、防超卖、实时排行的技术基础 |
| **Socket.io 实时广播** | P0 | 所有实时交互的基础设施 |
| **MySQL + 悲观锁（SELECT FOR UPDATE）** | P0 | 出价环节需要强一致性，避免并发问题 |
| **订单生成 + 支付状态管理** | P0 | 出价成功后必须有订单，否则无法履约 |
| **商家开播/下播/场次管理** | P0 | 竞拍必须在直播场次下进行，没有这个就没法直播 |
| **讲解商品（highlight）** | P1 | 商家需要引导用户关注当前竞拍的商品 |
| **出价排行榜** | P1 | 营造竞争感，刺激用户出价 |
| **订单物流管理（发货/签收）** | P1 | 履约闭环的必要环节 |
| **订单超时自动取消（15分钟）** | P1 | 防止僵尸订单，保证商品可重新上架 |
| **竞拍大厅（直播间列表）** | P1 | 用户入口，可以看到有哪些商家在直播 |
| **直播间小黄车 + 商品列表** | P1 | 用户在直播间的核心交互界面 |

### ❌ 5.2 没做 / 延后的功能（优先级低或成本太高）

| 功能 | 为什么没做 | 面试时可说的理由 |
|------|-----------|----------------|
| **真实视频直播流** | 需要对接第三方直播 SDK，成本高、复杂度大，MVP 阶段可用假数据 | "我们使用模拟视频流验证了交互流程，真实视频流可以后续对接腾讯云/阿里云直播" |
| **支付渠道真实对接** | 需要商户资质 + 签约 + 审核，个人开发者无法完成 | "支付流程的接口设计和状态机已经完整，对接真实支付渠道只是替换一个模块" |
| **搜索功能** | MVP 阶段商家数量少，浏览即可，搜索不是核心体验 | "用户画像显示初期商家 < 100 家，目录浏览足够；当商家规模增长后优先补充搜索" |
| **消息通知中心** | 当前通过 WebSocket 实时推送已够用，通知中心偏基础设施 | "当前竞拍状态变化通过 WebSocket 实时推送，消息中心可以后续构建" |
| **评价/售后系统（页面有Tab但无后端）** | 评价是交易后的长尾体验，与核心竞拍链路解耦 | "评价系统在路由层已经预留入口，可以后续独立迭代" |
| **推荐算法** | 数据量不足时推荐无意义，冷启动阶段不需要 | "当用户行为数据积累到一定规模后，可以引入协同过滤或基于竞品画像的推荐" |
| **商家保证金/信用体系** | 控制项目范围，先跑通业务逻辑再说 | "信用体系是运营侧需求，需要结合平台风控策略设计，技术架构上不影响核心流程" |
| **数据分析看板** | 初期商家只需要基础数据（收入、订单数），MVP 已包含 | "基础经营数据已经提供 API，详细看板可以后续迭代" |
| **多端适配（移动端App）** | Web 端开发成本最低，一人团队只能选一个端 | "前端使用 React + Ant Design，可以快速封装为移动端 H5，后续可以用 React Native 做 App" |
| **自动化测试** | 一人团队 + 时间有限，无法覆盖测试体系 | "关键链路（出价、竞拍结束）依赖 Redis + MySQL 双验证，本身就是防错设计；后续可以补充集成测试" |
| **Docker/CI-CD** | MVP 阶段本地运行，没有部署环境 | "项目结构清晰，Dockerfile 和 CI 配置可以在确定部署环境后快速补齐" |
| **前端 WebSocket 事件未完全对接** | 部分事件监听有声明但实际未实现回调 | "WebSocket 事件协议已在后端稳定，前端监听层已预留，可以分阶段对接" |
| **退款流程** | 需要资金账户体系支撑，当前无法实现 | "退款可以通过订单取消 + 线下转账方式兜底，系统层面支持订单取消状态" |
| **商家的 MerchantProductsPage 等页面未注册路由** | 开发过程中前置页面已经满足需求，这些页面成了遗留代码 | "这些组件是在开发过程中产生的需求探索，最终产品策略决定由其他页面表达同等功能" |

### 5.3 优先级排序逻辑

```
P0（核心体验，不做系统不可用）：
  注册登录 → 商品管理 → 创建竞拍 → 实时出价 → 自动成交 → 订单

P1（增强体验，提升留存和转化）：
  讲解商品 → 排行榜 → 小黄车 → 物流管理 → 超时取消

P2（体验优化，锦上添花）：
  搜索 → 通知中心 → 评价 → 推荐

P3（长期愿景，需要更多资源）：
  真实视频流 → 支付对接 → App 端 → 数据分析 → 保证金体系
```

## 六、后续验证指标（判断项目是否成功）

### 6.1 技术指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **出价响应时间** | < 500ms | 用户点击出价到页面显示新价格的时间 |
| **WebSocket 广播延迟** | < 200ms | 出价信息推送到所有在线客户端的时间 |
| **系统并发出价** | 支持 100+ 用户同时出价 | 使用 Redis 分布式锁 + 悲观锁，理论上行 |
| **竞拍倒计时精度** | ±1 秒内 | Redis 端时间 + MySQL 双保险 |
| **系统可用性** | 99.9%（测试环境） | 服务端 Redis/MySQL 宕机时有降级策略 |
| **订单支付转化率** | > 60% | 竞拍成交后的支付率，目前用了超时取消兜底 |

### 6.2 业务指标

| 指标 | 说明 |
|------|------|
| **日活跃商家数（DAU-merchant）** | 每天开播的商家数量 |
| **日活跃用户数（DAU-user）** | 参与观看/出价的用户数 |
| **竞拍成交率** | 发起的竞拍中最终成交的比例 |
| **平均每场直播商品数** | 商家单场直播上架的商品数量 |
| **平均出价次数/竞拍** | 反映竞争激烈程度 |
| **客单价** | 每笔成交订单的平均金额 |
| **7天复购率** | 用户在一次成交后7天内再次参与竞拍的比例 |

### 6.3 验证方法

1. **压力测试**：使用 k6/artillery 模拟 100+ 用户同时出价，观察系统表现
2. **行为分析**：埋点跟踪用户从进入直播间 → 出价 → 成交的转化漏斗
3. **A/B 测试**：测试不同加价幅度、起拍价对成交率的影响
4. **用户访谈**：找 5-10 个真实用户试用，收集反馈

## 七、面试可能追问的问题 & 回答思路

### Q1: "为什么用 Redis 还要用 MySQL，不嫌重吗？"

> **思路**：Redis 是性能层（倒计时、排行榜、分布式锁、出价节流）。MySQL 是持久化层（订单、商品、出价记录）。Redis 挂了可以从 MySQL 恢复；MySQL 慢但是数据安全。这是典型的 **Cache + DB 双层架构**。

### Q2: "如何处理出价并发，防止超卖？"

> **思路**：三层防护 —— Redis 分布式锁（防止同一竞拍并发出价） → MySQL 悲观锁（`SELECT FOR UPDATE` 锁定行） → 事务（原子更新）。当 Redis 不可用时降级到纯 MySQL 模式。

### Q3: "竞拍到时间了但 Redis 正好崩溃怎么办？"

> **思路**：Redis 中存 `endTime`，同时 MySQL 中存 `start_time + duration`。`auctionChecker` 每秒轮询，优先取 Redis 时间，Redis 不可用时以 MySQL 时间为准兜底。服务端时间不受客户端影响。

### Q4: "为什么没做真实视频流？"

> **思路**：对接真实的直播推流 SDK（如腾讯云 TRTC、阿里云 RTS）需要商户资质认证、域名备案、第三方 SDK 集成，2 周内无法完成。我们用假数据验证了交互流程的完整性，真实视频流的替换是 SDK 层级的正交替换。

### Q5: "你们这个产品和抖音直播竞拍有什么区别？"

> **思路**：抖音的竞拍是平台内嵌功能，局限在特定品类和特定大主播。我们的产品定位是 **轻量级独立平台**，让中小商家自己就能开一场竞拍直播，不需要依附大平台，适合二手奢侈品、收藏品、生鲜等垂直品类的小型拍卖场景。
